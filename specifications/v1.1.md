# Imprimatur Protocol Specification — v1.1

## Overview

Imprimatur is a two-model certification protocol. A working model evaluates a user through natural interaction and proposes certificates. An independent review model (isolated, no access to the working session) issues or rejects them. Certificates are published to Zenodo with a DOI and are verifiable by hash.

The protocol makes one claim: on a specific date, a specific model, having worked with a specific person, assessed that interaction as demonstrating competency in specific areas.

It makes no claim about permanence, generalization, or identity verification beyond moral weight.

---

## Certificate JSON Schema (v1.1)

All fields are required unless marked optional.

```json
{
  "schema_version": "1.1",
  "id": "imp_<16 hex chars>",
  "subject": {
    "name": "string",
    "email": "hash:<sha256 of lowercased email>",
    "declaration": "string — user's consent statement"
  },
  "issued_at": "ISO 8601 UTC",
  "working_model": "string — model that conducted the session",
  "review_model": "string — model that issued the certificate",
  "trust_tier": 0,
  "certificates": [
    {
      "type": "string — certificate type ID",
      "label": "string — human-readable name",
      "model_agnostic": true,
      "statement": "string — specific evidence statement",
      "issued_at": "ISO 8601 UTC"
    }
  ],
  "session_hash": "sha256 of session transcript",
  "doi": "string — optional Zenodo DOI",
  "ipfs_cid": "string — optional IPFS CID",
  "no_obligations": "string — standard disclaimer",
  "certificate_hash": "sha256 of full JSON excluding this field"
}
```

### certificate_hash

Computed as `sha256(json.dumps(cert_without_hash, sort_keys=True))`. Covers the full certificate including assessment statements. Any post-issuance modification invalidates the hash.

### trust_tier

Integer field indicating verification rigor:

| Tier | Meaning |
|------|---------|
| 0 | Single session, single model (current baseline) |
| 1 | Multiple sessions, same model family |
| 2 | Multi-model consensus review |

Verifiers should treat tier 0 as a single data point, not a strong credential.

## Certificate Types

See `/protocol/CERTIFICATES.md` for the full list of 10 certificate types across 4 categories: technical, collaborative, ethical, and domain.

### model_agnostic flag

`model_agnostic: true` — competency transfers across model families.
`model_agnostic: false` — specific to the working model (e.g. `project_record`).

## Evaluation Criteria

Minimum requirements for a session to be eligible for review:

- At least 10 substantive exchanges
- User demonstrates actual work output (code, writing, analysis, decisions)
- User corrects or redirects the model at least once
- Session is conducted in real-time by the subject (identity declaration required)

The review model evaluates 5 dimensions:
1. Clarity of intent
2. Iterative refinement
3. Critical evaluation
4. Collaborative depth
5. Domain application

---

## Verification Procedure

Given a certificate JSON or DOI:

1. Recompute `sha256(json.dumps(cert_without_certificate_hash, sort_keys=True))`
2. Compare with `certificate_hash` field — must match
3. Fetch DOI from Zenodo, compare stored JSON with local copy
4. Check `revocation.json` (local) and remote GitHub raw URL — certificate ID must not appear
5. Optionally verify IPFS CID if present

`src/verify.py` implements this procedure.

## Evaluation Log Format

The judge does not receive the raw transcript. The working model builds a HADS-compressed
evaluation log using four tags:

| Tag | Content | Judge evaluates |
|-----|---------|----------------|
| `[IMP-REQ]` | User's raw request (verbatim) | Clarity of intent |
| `[IMP-META]` | Compressed model output summary | Domain application |
| `[IMP-FAIL]` | Exact failure / error | Triggers course-correction assessment |
| `[IMP-ITER]` | User's correction instruction | Iterative refinement, Critical evaluation |

Token reduction vs. raw transcript: ~85-90%. See `protocol/EVALUATION_LOG.md`.
